# Aurora 项目-开票-回款管理系统需求文档

## 1. 文档目标

本文档用于定义 Aurora 后端 v1 的业务需求，作为后续接口设计、数据库设计和开发实现的依据。

系统定位：
- 公司内部使用的轻量级管理系统。
- 主要用于管理项目、开票、回款三类核心业务数据。
- 第一阶段仅建设后端，前后端分离。
- 后续前端可作为静态资源打包进后端统一运行。

适用范围：
- 面向 1-3 名公司内部成员。
- 不考虑多租户、复杂组织架构、审批流、附件、消息通知、审计等能力。

## 2. 建设目标

### 2.1 业务目标

建立一个最小可用的内部系统，满足以下诉求：
- 管理项目基础信息。
- 记录项目对应的开票信息。
- 记录项目对应的回款信息。
- 在项目维度查看实时汇总的已开票金额和已回款金额。
- 通过登录和角色授权控制不同人员的功能访问范围。
- 支持管理员维护系统账号。

### 2.2 非目标

以下内容不属于 v1 范围：
- 客户主数据管理。
- 合同附件上传与归档。
- 发票号、回款流水号、备注说明等扩展字段。
- 审批、作废、撤销、状态流转。
- 月报、季报、复杂统计报表。
- 消息通知、待办提醒。
- 操作审计日志。
- 多公司、多组织、多租户能力。

## 3. 用户与角色

### 3.1 用户体系

系统采用账号密码登录。

规则如下：
- 每个账号具有唯一登录名。
- 每个账号可以绑定多个角色。
- 账号支持启用和停用。
- 停用账号不可登录系统。
- 管理员可以重置其他账号密码。

### 3.2 角色定义

系统固定三类角色：

#### 管理员
- 管理系统账号。
- 分配账号角色。
- 查看和维护全部业务数据。

#### 项目管理
- 查看项目信息。
- 新增、编辑项目。
- 查看和维护开票信息。
- 查看和维护回款信息。

#### 财务
- 查看项目信息。
- 查看和维护开票信息。
- 查看和维护回款信息。

### 3.3 权限原则

第一版采用纯角色权限模型：
- 不做数据隔离，不区分“只能看自己项目”。
- 一个用户拥有多个角色时，权限按角色并集生效。

## 4. 核心业务对象

### 4.1 项目

项目是系统的核心主对象，字段如下：
- 名称
- 客户
- 合同号
- 签约时间
- 合同金额

项目包含两个虚拟汇总字段：
- 已开票金额
- 已回款金额

说明：
- 客户字段为普通文本，不建立独立客户表。
- 已开票金额由该项目下所有开票记录金额求和得到。
- 已回款金额由该项目下所有回款记录金额求和得到。
- 汇总字段不单独存库，以实时统计结果返回。

### 4.2 开票

开票记录字段如下：
- 所属项目
- 金额
- 开票时间

说明：
- 一个项目可以有多条开票记录。
- 开票金额必须大于 0。
- 开票时间必填。

### 4.3 回款

回款记录字段如下：
- 所属项目
- 金额
- 回款时间

说明：
- 一个项目可以有多条回款记录。
- 回款金额必须大于 0。
- 回款时间必填。

## 5. 业务流程

### 5.1 登录与进入系统

基础流程：
1. 用户输入账号和密码登录。
2. 系统校验账号状态和密码。
3. 登录成功后建立登录态，并返回当前用户角色信息。
4. 前端依据角色显示可访问功能。

### 5.2 项目录入与维护

基础流程：
1. 具备权限的用户创建项目。
2. 填写名称、客户、合同号、签约时间、合同金额。
3. 后续可编辑项目基础信息。
4. 项目列表和详情页面可查看实时汇总金额。

### 5.3 开票录入与维护

基础流程：
1. 具备权限的用户进入某个项目。
2. 新增一条开票记录，填写金额和开票时间。
3. 系统保存后，项目的已开票金额实时更新。
4. 已存在的开票记录可编辑和删除。

### 5.4 回款录入与维护

基础流程：
1. 具备权限的用户进入某个项目。
2. 新增一条回款记录，填写金额和回款时间。
3. 系统保存后，项目的已回款金额实时更新。
4. 已存在的回款记录可编辑和删除。

### 5.5 用户管理

管理员可完成以下操作：
- 查看账号列表。
- 新增账号。
- 编辑账号基础信息。
- 启用或停用账号。
- 重置账号密码。
- 为账号分配一个或多个角色。

## 6. 页面与功能清单

### 6.1 登录页

功能点：
- 账号输入。
- 密码输入。
- 登录提交。
- 登录失败提示。

### 6.2 首页或项目列表页

功能点：
- 查看项目列表。
- 按名称、客户、合同号查询项目。
- 查看项目汇总字段。
- 进入项目详情。
- 创建项目。

### 6.3 项目详情页

功能点：
- 查看项目基础信息。
- 查看已开票金额、已回款金额。
- 查看开票记录列表。
- 查看回款记录列表。
- 编辑项目。
- 新增、编辑、删除开票记录。
- 新增、编辑、删除回款记录。

### 6.4 用户管理页

功能点：
- 查看用户列表。
- 新增用户。
- 编辑用户。
- 启用/停用用户。
- 重置密码。
- 分配角色。

## 7. 业务规则

### 7.1 通用规则

- 所有金额字段必须大于 0。
- 所有日期字段必须填写。
- 所有必填文本字段不能为空。
- 所有创建和修改操作都需要登录后执行。

### 7.2 项目规则

- 项目名称必填。
- 客户必填。
- 合同号必填。
- 签约时间必填。
- 合同金额必填且必须大于 0。
- 项目列表和详情需返回实时汇总的已开票金额、已回款金额。

### 7.3 开票与回款规则

- 开票和回款必须关联一个已存在的项目。
- 项目不存在时不允许创建关联记录。
- 修改开票或回款记录后，项目汇总金额需立即反映最新结果。

### 7.4 删除规则

- 开票记录可删除。
- 回款记录可删除。
- 项目在不存在任何开票和回款记录时允许删除。
- 项目存在任意开票或回款记录时，不允许直接删除，并返回明确提示。

## 8. 非功能要求

### 8.1 易用性

- 业务模型保持最小化，减少学习成本。
- 接口命名和错误提示应清晰直观。

### 8.2 可维护性

- 后端接口应保持稳定，便于后续接入独立前端。
- 权限模型和用户模型应支持未来平滑扩展。

### 8.3 部署要求

- 第一版以单体后端服务运行。
- 数据库使用 SQLite，便于开发、演示和单机部署。
- 后续应允许切换到正式关系型数据库而不影响业务模型。

## 9. 验收标准

系统至少满足以下验收项：
- 可通过账号密码登录系统。
- 停用账号无法登录。
- 管理员可新增账号、停用账号、重置密码、分配角色。
- 可新增、编辑、删除项目。
- 可为项目新增、编辑、删除开票记录。
- 可为项目新增、编辑、删除回款记录。
- 项目详情和列表可正确显示已开票金额和已回款金额。
- 项目存在关联开票或回款记录时不可删除。
- 不同角色访问受限功能时，系统可进行权限控制。
